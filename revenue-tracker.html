What this page shows & where the data comes from

This tracker joins two independent views of the same revenue. The forecast is computed live, on every page load, from the Pipeline's schedule: each project's EstimatedProjectValue divided by its construction-phase duration gives a weekly revenue rate, spread across wherever that phase actually sits on the calendar. Nothing is stored — move a bar in the Pipeline and this forecast moves with it, so the two tools can never disagree. Solid gold is CA-signed work; the striped portion is projected (DA-signed, unsigned, or paused).

The actuals come from the monthly WIP workbook that Joey and Ryan maintain on the Operations site — the percent-complete reviews that turn QuickBooks costs into earned revenue per job. "Sync from WIP" reads that workbook directly from SharePoint, takes Ryan's (RS) sheets as the record with Joey's (JW) percent-complete stored alongside for comparison, and writes one row per job per month to the CORE_Revenue list. The display always reads from CORE_Revenue, never from the workbook itself, so the page stays fast and the history survives workbook reorganizations.

In short: CORE_Projects owns what will happen (changes are keyed in the Pipeline only); CORE_Revenue owns what did happen (written only by WIP sync, frozen thereafter); this page is where they meet. The Δ flags in the project table mark jobs where the WIP contract value has drifted from the Pipeline's — usually a change order keyed into one system but not the other.

Recommended next steps for the architecture: move CORE_Revenue to OperationsSecure once the visibility question above is settled, since costs-vs-contract exposes job margin; add the CORE JobCode to the WIP workbook's project rows so name-matching becomes deterministic instead of fuzzy; stand up a shared auth redirect page so each new CORE tool stops requiring its own Azure registration; build the change-order process so contract-value drift is prevented rather than flagged; and when a QuickBooks API plan materializes, it replaces the sync step here — nothing downstream changes.
